System Architecture

Four subsystems, two operating systems, one loopback address. SITL supplies real flight physics; FlightGear only renders. A screen-captured frame feed and an independent MAVLink thread meet at the point of detection.

Critical Bug — The AI Was Watching the Desktop Root-caused
mss capture region · uncalibrated
The single most consequential defect in the project: an uncalibrated screen-capture region caused YOLO to run against desktop wallpaper and folder icons instead of FlightGear's 3D scene — producing false "car" detections with no relationship to the simulated flight at all.
Fig. — Confirmed false positives: detection boxes on desktop icons, unrelated to the simulated drone's view.
05
Capture Region Calibrated & Verified Fixed
pygetwindow · geometry-checked · live preview
Fixed by programmatic window geometry verification, a mandatory pre-flight capture_region_check.jpg, and — after this log — a live preview mode with fail-loudly bounds checking, rather than silently trusting stale window coordinates.
Fig. — FlightGear (left) and the live preview capture (right) showing matching content — the region is finally correct.
06
Sync Regression — Ghost Process on Port 5503 Root-caused
fgfs.exe (minimized) · stale FDM binding
A second, minimized FlightGear instance silently held the UDP FDM port, leaving a freshly opened window frozen at its startup pose while SITL genuinely flew — diagnosed by cross-checking FlightGear's own HUD altitude against MAVProxy's console.
Fig. — Diagnostic evidence: FlightGear HUD static at 0 while Console genuinely reports 98 m altitude.
07
Telemetry Fusion & Orchestration Operational
pymavlink thread · UDP :14551 · launch_drone_sim.bat
A background MAVLink thread now feeds live lat/lon/alt into every detection row without ever blocking frame capture. A single hardened .bat launcher sequences FlightGear → SITL → detection with port checks and a mandatory human sync-verification checkpoint before recording begins.
